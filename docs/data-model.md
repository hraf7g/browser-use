# Data Model: UAE Tender Watch (UTW) v1

## 1. Purpose

This document defines the minimum viable database model for UAE Tender Watch v1.

The schema is designed to support:

- user accounts
- one keyword profile per user
- exactly 2 monitored sources in v1
- tender ingestion with safe deduplication
- keyword-based matching
- crawl run tracking
- daily email delivery tracking

This schema is intentionally small. It should be easy to implement, migrate, and reason about.

---

## 2. Design Rules

The schema follows these rules:

- keep only tables required for v1
- prefer explicit fields over vague JSON blobs
- enforce deduplication at the database level where possible
- allow for incomplete source data when portals do not expose stable identifiers
- separate tender storage from delivery state
- keep operational tracking simple

---

## 3. Core Tables

## 3.1 `users`

Represents customer accounts.

Required fields:

- `id` — UUID primary key
- `email` — string, required
- `password_hash` — string, required
- `is_active` — boolean, required, default true
- `created_at` — timestamp, required
- `updated_at` — timestamp, required

Optional fields:

- `last_login_at` — timestamp, nullable

Constraints:

- unique index on `email`

Notes:

- v1 uses one account per person
- no team/org model in v1

---

## 3.2 `keyword_profiles`

Represents the user’s matching preferences.

Required fields:

- `id` — UUID primary key
- `user_id` — UUID foreign key to `users.id`
- `keywords` — text or text array, required
- `alert_enabled` — boolean, required, default true
- `created_at` — timestamp, required
- `updated_at` — timestamp, required

Optional fields:

- `industry_label` — string, nullable

Constraints:

- unique constraint on `user_id`

Notes:

- v1 supports one keyword profile per user
- keywords should be stored in a simple implementable format, such as text array or normalized text list
- do not overdesign this into a many-table taxonomy model in v1

---

## 3.3 `sources`

Represents monitored tender portals.

Required fields:

- `id` — UUID primary key
- `name` — string, required
- `base_url` — string, required
- `status` — enum or constrained string, required
- `created_at` — timestamp, required
- `updated_at` — timestamp, required

Optional fields:

- `last_successful_run_at` — timestamp, nullable
- `last_failed_run_at` — timestamp, nullable
- `failure_count` — integer, required, default 0
- `notes` — text, nullable

Suggested status values:

- `healthy`
- `degraded`
- `broken`

Constraints:

- unique constraint on `name`
- unique constraint on `base_url`

Notes:

- v1 seeds exactly 2 records
- this table is operational and configuration-oriented

---

## 3.4 `tenders`

Stores normalized tender records discovered from source listing pages.

Required fields:

- `id` — UUID primary key
- `source_id` — UUID foreign key to `sources.id`
- `source_url` — string, required
- `title` — string, required
- `issuing_entity` — string, required
- `closing_date` — timestamp, required
- `dedupe_key` — string, required
- `created_at` — timestamp, required
- `updated_at` — timestamp, required

Optional fields:

- `tender_ref` — string, nullable
- `opening_date` — timestamp, nullable
- `category` — string, nullable
- `ai_summary` — text, nullable
- `published_at` — timestamp, nullable

Constraints:

- unique constraint on `(source_id, tender_ref)` where `tender_ref` is not null
- unique constraint on `(source_id, dedupe_key)`

Notes:

- some portals may not expose a stable tender reference number
- `dedupe_key` exists specifically for those cases
- `source_url` should not be globally unique because the same portal may present unstable or repeated URLs
- the database must not assume every tender has a durable public identifier

### `dedupe_key` guidance

The `dedupe_key` should be deterministic and generated by application logic.

Example inputs may include:

- source identifier
- normalized title
- normalized issuing entity
- normalized closing date
- source URL when useful

The exact hash recipe belongs in implementation, but the rule is:

- stable enough to catch repeat crawls of the same listing
- conservative enough to avoid collapsing clearly distinct tenders

---

## 3.5 `tender_matches`

Stores the relationship between a user and a tender that matched the user’s keyword profile.

Required fields:

- `id` — UUID primary key
- `user_id` — UUID foreign key to `users.id`
- `tender_id` — UUID foreign key to `tenders.id`
- `created_at` — timestamp, required

Optional fields:

- `matched_keywords` — text or text array, nullable
- `sent_at` — timestamp, nullable

Constraints:

- unique constraint on `(user_id, tender_id)`

Notes:

- this table is the core duplicate-alert prevention layer
- a record should exist only once per user/tender pair
- `sent_at` marks whether that match has already been included in a successful user email

---

## 3.6 `crawl_runs`

Tracks each crawl attempt for each source.

Required fields:

- `id` — UUID primary key
- `source_id` — UUID foreign key to `sources.id`
- `status` — enum or constrained string, required
- `started_at` — timestamp, required
- `finished_at` — timestamp, nullable
- `created_at` — timestamp, required

Optional fields:

- `new_tenders_count` — integer, nullable
- `failure_reason` — text, nullable
- `failure_step` — string, nullable
- `run_identifier` — string, nullable

Suggested status values:

- `running`
- `success`
- `failed`

Notes:

- keep this table operational only
- detailed enough to debug failures
- simple enough to query in the operator page
- a separate `crawl_errors` table is not required in v1 if `crawl_runs` already captures useful failure detail

---

## 3.7 `email_deliveries`

Tracks each daily brief attempt per user.

Required fields:

- `id` — UUID primary key
- `user_id` — UUID foreign key to `users.id`
- `delivery_type` — enum or constrained string, required
- `status` — enum or constrained string, required
- `attempted_at` — timestamp, required
- `created_at` — timestamp, required

Optional fields:

- `sent_at` — timestamp, nullable
- `match_count` — integer, nullable
- `failure_reason` — text, nullable

Suggested `delivery_type` values:

- `daily_brief`
- `founder_failure_alert`

Suggested `status` values:

- `queued`
- `sent`
- `failed`

Notes:

- this table should store minimum useful delivery history
- it does not need to store a full JSON snapshot of all included tenders in v1
- the source of truth for what has been sent remains the `sent_at` state on `tender_matches`

---

## 4. Required Indexes

Only keep indexes that clearly matter in v1.

## `users`
- unique index on `email`

## `keyword_profiles`
- unique index on `user_id`

## `sources`
- unique index on `name`
- unique index on `base_url`

## `tenders`
- index on `source_id`
- index on `closing_date`
- index on `created_at`
- unique partial index on `(source_id, tender_ref)` where `tender_ref` is not null
- unique index on `(source_id, dedupe_key)`

## `tender_matches`
- unique index on `(user_id, tender_id)`
- index on `(user_id, sent_at)`

## `crawl_runs`
- index on `source_id`
- index on `started_at`

## `email_deliveries`
- index on `user_id`
- index on `attempted_at`

---

## 5. Data Retention Guidance

Keep retention simple.

## `tenders`
Retain for at least 12 months in v1 so users can review past opportunities.

## `tender_matches`
Retain as long as the corresponding tender is retained.

## `crawl_runs`
Retain at least 60 to 90 days for debugging and operational review.

## `email_deliveries`
Retain at least 90 days for support and troubleshooting.

---

## 6. Relationships Summary

- one `user` has one `keyword_profile`
- one `source` has many `tenders`
- one `source` has many `crawl_runs`
- one `tender` can match many users
- one `user` can match many tenders
- one `user` can have many `email_deliveries`

---

## 7. v1 Boundary Notes

This schema does **not** include:

- organizations or teams
- per-user portal subscriptions beyond keyword filtering
- proposal drafts
- bid submissions
- document storage
- audit/event streams
- advanced analytics tables

Those are intentionally out of scope for v1.

---

## 8. Final v1 Schema Philosophy

The most important database rules are:

1. **dedupe tenders safely**
2. **prevent duplicate user alerts**
3. **track crawl health simply**
4. **keep the schema small enough to implement correctly**

That is enough for a solid first production version.
